Signal Forge Output Gallery

What You'll Learn: Understand what Signal Forge generates for each content type and mode, so you can choose the right format for your needs and recognize quality outputs.

Quick Start: Understanding Your Output

Run Signal Forge with any content type to see its output:

signal-forge generate --type deck --input "modernize legacy platform"

Check the voice score in the console output:

✓ Content generated successfully
  Voice Score: 8.2/10 (Executive Advisory)
  Output: output/strategic-modernization-deck.pptx

Voice scores above 7.0 are ready to use. Scores below 5.0 indicate the wrong mode was detected.

Executive Advisory Outputs

Strategic Deck

Generate a strategy presentation:

signal-forge generate \
  --type deck \
  --format pptx \
  --input "cloud migration strategy for retail company"

Expected output structure:

Slide 1:  Cover (title, date, confidential marking)
Slide 2:  Executive Summary
Slide 3:  Current State Assessment
Slide 4:  Key Challenges
Slide 5:  Strategic Approach
Slide 6:  Phase 1 Details
Slide 7:  Phase 2 Details
Slide 8:  Investment & Timeline
Slide 9:  Risk Mitigation
Slide 10: Next Steps

Voice characteristics you'll see:

  • Confident recommendations: "I recommend a three-phase approach..."
  • Pattern recognition: "We've seen this work across similar transformations..."
  • Business outcomes: "This delivers 40% cost reduction while enabling 3x faster launches..."
  • Scannable bullets: Short, declarative statements, not paragraphs

Sample slide content:

RECOMMENDATION: PHASED CLOUD MIGRATION

I recommend a three-phase approach based on
successful patterns from retail transformations:

Phase 1: Foundation (Q1-Q2)
• Establish AWS infrastructure
• Migrate dev/test environments
• Build cloud competency

Phase 2: Core Systems (Q3-Q4)
• E-commerce platform migration
• Customer data consolidation
• API gateway implementation

Phase 3: Optimization (Q1 Year 2)
• Auto-scaling and cost optimization
• Advanced analytics
• AI/ML capabilities

This approach minimizes disruption while
delivering early wins.

Format options:

signal-forge generate --type deck --format html --input "..."

# PDF (print-ready)

Architecture Document

signal-forge generate \
  --type architecture \
  --format html \
  --input "microservices e-commerce platform with event streaming"

Expected output structure (arc42 standard):

1.  Introduction and Goals
2.  Constraints
3.  Context and Scope
4.  Solution Strategy
5.  Building Block View
    - System Context
    - Container View
    - Component View
6.  Runtime View
7.  Deployment View
8.  Crosscutting Concepts
9.  Architecture Decisions
10. Quality Requirements
11. Risks and Technical Debt

Voice characteristics you'll see:

  • Precise technical language: "The API Gateway routes requests based on path prefix to backend services"
  • No provisional statements: "The system uses PostgreSQL" not "We might use PostgreSQL"
  • Implementation-ready: Specific technologies, versions, configurations
  • Diagram descriptions: Text descriptions of each architectural view

Sample section:

## 5. BUILDING BLOCK VIEW

### 5.1 Container View

The system consists of four primary containers:

**API Gateway**
- Technology: Kong 3.x on Kubernetes
- Responsibilities: Request routing, rate limiting,
  authentication
- Dependencies: Identity Service for JWT validation
- Scaling: Horizontal auto-scaling, 2-10 pods

**Commerce Service**
- Technology: Node.js 20 with TypeScript
- Responsibilities: Product catalog, pricing, inventory
- Database: PostgreSQL 15 with read replicas
- Events Published: OrderCreated, InventoryUpdated
- API: REST with OpenAPI 3.0 specification

**Order Service**
- Technology: Java 17 with Spring Boot 3.1
- Responsibilities: Order lifecycle management
- Database: PostgreSQL 15 with read replicas
- Events Consumed: OrderCreated
- Events Published: OrderShipped, OrderCancelled

**Notification Service**
- Technology: Python 3.11 with FastAPI
- Responsibilities: Email, SMS, push notifications
- Dependencies: Consumes events from Commerce and
  Order services
- External APIs: SendGrid, Twilio

Architecture Decision Record (ADR)

Generate a technical decision record:

signal-forge generate \
  --type adr \
  --format md \
  --input "choosing between Kafka and RabbitMQ for event streaming"

Expected output structure:

1. Title (ADR-NNN: Decision Title)
2. Status (Proposed/Accepted/Deprecated/Superseded)
3. Context (Why this decision is needed)
4. Decision (What was decided)
5. Consequences
   - Positive
   - Negative
6. Alternatives Considered
7. References

Voice characteristics you'll see:

  • Factual, not persuasive: States facts, not opinions
  • Balanced consequences: Lists both pros and cons
  • Clear rationale: Explains the "why" with data
  • Linked decisions: References related ADRs

Sample ADR:

# ADR-003: Event Streaming Platform Selection

## Status
Accepted

## Context
The microservices architecture requires an event
streaming platform for asynchronous communication.

Requirements:
- Throughput: 100,000 messages/second
- Durability: Events must persist for replay
- Ordering: Strict ordering per partition
- Budget: $50K/year maximum
- Team: Moderate Kafka experience exists

## Decision
Use Apache Kafka deployed via Strimzi operator on
existing Kubernetes cluster.

## Consequences

### Positive
- High throughput meets requirements with 3x margin
- Event replay enables event sourcing patterns
- Strong monitoring ecosystem (Prometheus, Grafana)
- Team has prior experience
- No vendor lock-in

### Negative
- Operational complexity higher than managed services
- Requires 0.5 FTE dedicated ops capacity
- Learning curve for advanced features (KSQL, Streams)
- Self-managed upgrades and patching

## Alternatives Considered

### AWS MSK (Managed Kafka)
**Rejected**: Cost $120K/year at projected volume.
Vendor lock-in concerns for multi-cloud strategy.

### RabbitMQ
**Rejected**: Throughput ceiling 50K messages/second
insufficient for growth projections.

### AWS SQS + SNS
**Rejected**: No native event replay capability.
Ordering guarantees insufficient for financial
transactions.

## References
- Performance benchmarks: [link to internal doc]
- Cost analysis: [link to spreadsheet]
- Related: ADR-001 (Cloud Platform Selection)

# PDF (archival)

Thought Leadership Outputs

Point of View (POV)

Generate an opinion piece:

signal-forge generate \
  --type pov \
  --format html \
  --input "why enterprise AI projects fail in POC phase"

Expected output structure:

1. Opening Hook (question or tension)
2. Personal Observation
3. Exploration of the Idea
4. Complications or Objections
5. Provisional Conclusion

Voice characteristics you'll see:

  • Question-led opening: "Why do enterprise AI projects keep dying in POC purgatory?"
  • Personal grounding: "I've watched this pattern repeat across a dozen organizations..."
  • Self-interrogation: "I used to think... Now I'm not so sure."
  • Provisional ending: "So here's where I've landed—for now."
  • Natural rhythm: Intentional fragments and varied sentence length

Sample opening:

Why do enterprise AI projects keep dying in POC purgatory?

I've watched this pattern repeat across a dozen
organizations now. The demo dazzles. Executives applaud.
Budget flows.

And then... nothing.

Six months later, the prototype sits abandoned while
the team moves on to the next shiny thing.

I used to think this was a technology problem. Get the
models right. Nail the infrastructure. Build the pipeline.

Now I'm not so sure.

Sample closing:

So here's where I've landed—for now.

Enterprise AI success isn't about the model. It's about
the muscle. The organizational muscle to actually change
how work gets done.

That's harder to demo in a boardroom. But it's the only
thing that actually works.

I could be wrong. I often am. But if you're about to
greenlight another AI POC, maybe pause and ask: do we
have the muscle to actually use this?

Understanding Voice Scores

Check voice scores in console output:

✓ Content generated successfully
  Voice Score: 8.2/10 (Executive Advisory)
  Confidence: High

Voice Score Ranges

Score Meaning Action
9.0-10.0 Excellent Ready to use as-is
7.0-8.9 Good Ready for most uses, minor polish optional
5.0-6.9 Acceptable Usable with review, consider --iterate
3.0-4.9 Weak Significant issues, likely wrong mode detected
0.0-2.9 Poor Re-generate with clearer input context

Improve low scores with iteration:

# Run with iteration for important content
signal-forge generate \
  --type deck \
  --input "cloud strategy" \
  --iterate 3

This generates 3 versions and selects the highest-scoring output.

## SOLUTION STRATEGY

The modernization program employs a strangler fig pattern
to incrementally replace the legacy monolith with a
microservices architecture.

Key architectural decisions:

1. **API Gateway Pattern**
   - Kong Gateway routes traffic between legacy and new services
   - Enables gradual migration without big-bang cutover
   - Provides unified authentication and rate limiting

2. **Event-Driven Architecture**
   - Apache Kafka for inter-service communication
   - Loose coupling enables independent deployment
   - Event sourcing for order and inventory domains

3. **Domain-Driven Design**
   - Bounded contexts: Catalog, Order, Customer, Payment
   - Each context owns its data
   - No shared databases between services

4. **Container Orchestration**
   - Kubernetes for deployment and scaling
   - Horizontal pod autoscaling based on CPU and custom metrics
   - Blue-green deployments for zero-downtime updates

Troubleshooting

Low Voice Scores (Below 5.0)

Problem: Generated content has voice score below 5.0

Likely causes:

  1. Input context is ambiguous
  2. Wrong mode auto-detected
  3. Conflicting signals in input

Solutions:

# Explicitly specify the mode
signal-forge generate \
  --type deck \
  --mode advisory \
  --input "your context here"

# Provide more specific context
signal-forge generate \
  --type deck \
  --input "Executive presentation recommending cloud migration for retail company with $50M revenue"

# Use iteration to generate multiple versions
signal-forge generate \
  --type deck \
  --input "your context" \
  --iterate 3

Wrong Content Type Generated

Problem: Generated a POV when you wanted a deck

Cause: Content type not specified or wrong type used

Solution:

# Always specify --type explicitly
signal-forge generate --type deck --input "..."

# Verify type before generating
signal-forge list-types

Available types:

  • deck - Strategy presentation
  • brief - Executive summary
  • roadmap - Initiative timeline
  • architecture - Technical documentation
  • adr - Architecture decision record
  • spec - Technical specification
  • pov - Opinion piece (800-1200 words)
  • paper - In-depth analysis (3000-8000 words)
