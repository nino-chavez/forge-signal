Signal Forge Use Cases Guide

Learn how to transform rough notes, meeting recaps, and research into polished executive presentations, technical documentation, and thought leadership content using Signal Forge.

Quick Start

Generate your first document in 60 seconds:

# 1. Create a simple input file
echo "# Meeting Notes
- Discussed cloud migration strategy
- Budget approved: $2M
- Timeline: 6 months
- Key risk: team capacity" > notes.md

# 2. Generate an executive brief
forge generate brief --input notes.md --format pdf

# 3. Find your output
ls output/brief_*.pdf

Core Concepts

Input Files

Signal Forge works with markdown files containing your raw content—meeting notes, observations, technical decisions, or research findings.

Document Types

Signal Forge generates five document types:

  • Brief: Executive summaries and strategic documents
  • Deck: Presentation slide decks
  • Architecture: Technical architecture documentation
  • ADR: Architecture Decision Records
  • POV: Thought leadership and opinion pieces

Agentic Workflows

Add intelligence to generation with:

  • --research: Pull current web context
  • --iterate: Automatically refine until quality thresholds met
  • --memory: Maintain context across sessions

Executive Presentations

Use Case 1: Client Strategy Deck from Meeting Notes

What You'll Create: A polished 10-15 slide presentation from rough meeting notes.

Scenario: You've completed a strategy session and need a follow-up deck for stakeholders.

Step 1: Prepare Your Input

Create meeting-recap.md:

# Client Strategy Session - TechCorp
Date: November 25, 2024
Attendees: Sarah (CEO), Mike (CTO), Jennifer (Product)

## Key Discussion Points
- Current platform is aging, 7 years old
- Migration to cloud discussed but concerns about cost
- Customer complaints about slow performance increasing
- Competition launching new features quarterly
- Budget approved for Q1 modernization

## Decisions Made
- Phased migration approach preferred
- Start with customer-facing systems
- Need ROI analysis before full commitment

## Open Questions
- Build vs buy for new commerce engine?
- Timeline expectations from board?
- Staffing for parallel operations?

Step 2: Generate the Deck

forge generate deck --input meeting-recap.md --format pptx,html

Step 3: Review Output

Expected output structure:

Slide Content
1 Executive Summary
2-3 Current State Assessment
4-6 Proposed Migration Approach
7-9 Phased Timeline & Milestones
10-11 Investment Framework & ROI
12-13 Risk Considerations
14-15 Recommended Next Steps

Output Location:

output/
├── deck_meeting-recap_20241125_143022.pptx
└── deck_meeting-recap_20241125_143022.html

Technical Documentation

Use Case 3: Architecture Document from Design Session

What You'll Create: Comprehensive architecture documentation with diagrams and decision rationale.

Scenario: Document a new microservices architecture after a design session.

Step 1: Capture Design Session Notes

Create design-session-notes.md:

# Commerce Platform Redesign - Design Session

## Current Architecture
- Monolithic Java application
- PostgreSQL database
- Basic caching with Redis

## Proposed Architecture
- API Gateway for routing
- Separate services: Catalog, Cart, Checkout, Orders
- Event-driven communication via Kafka
- Each service owns its data

## Key Decisions
- Go with Kubernetes for orchestration
- Use GraphQL for frontend API
- PostgreSQL for transactional, MongoDB for catalog
- Auth via OAuth 2.0 + OIDC

## Non-functional Requirements
- 99.9% availability
- <200ms p95 latency
- Handle 10k concurrent users
- GDPR compliant

## Open Items
- Observability stack selection
- CI/CD pipeline design
- Data migration strategy

Step 2: Generate Architecture Documentation

forge generate architecture --input design-session-notes.md --mode architecture --format html,pdf

Expected Output Structure

The generated document includes:

Context & Scope Business drivers and system boundaries (C4 Level 1)
Container Architecture Service descriptions and interactions (C4 Level 2)
Component Details Internal structure of each service
Data Architecture Data flow and storage patterns
Deployment Infrastructure and orchestration
Security Authentication, authorization, data protection
ADRs Rationale for key technical decisions

Use Case 4: Architecture Decision Record

What You'll Create: A structured ADR documenting a key technical decision.

Scenario: Document why you chose Kafka over alternatives for event streaming.

Step 1: Document Decision Context

Create kafka-decision.md:

# Event Streaming Decision Context

## The Decision
Need to choose event streaming platform for new architecture

## Options Considered
1. Apache Kafka
2. RabbitMQ
3. AWS SQS + SNS

## Evaluation Criteria
- Throughput: Need 100k msgs/sec
- Durability: Events must persist for replay
- Ordering: Strict ordering required per partition
- Team expertise: Some Kafka experience
- Cost: Self-hosted preferred (already have K8s)

## Analysis
Kafka wins on throughput and durability
RabbitMQ easier but lower throughput
AWS SQS simpler but vendor lock-in

## Recommendation
Apache Kafka with Strimzi operator on K8s

Step 2: Generate ADR

forge generate adr --input kafka-decision.md --format markdown,html

Expected ADR Structure

**Status**: Accepted
**Date**: 2024-11-25
**Decision Makers**: Platform Architecture Team

## Consequences

### Positive
- High throughput (100k+ msgs/sec)
- Event replay capability
- Strong ordering guarantees

### Negative
- Operational complexity
- Steeper learning curve
- Resource overhead

## Alternatives Considered
[Detailed comparison table]

## References

Thought Leadership Content

Use Case 5: Industry Trend Point of View

What You'll Create: An 800-1200 word thought leadership piece exploring an industry trend.

Scenario: Write about the gap between AI hype and enterprise reality.

Step 1: Capture Your Observations

Create ai-observations.md:

# Observations on Enterprise AI Adoption

## What I'm Seeing
- Lots of POCs, few production deployments
- Executives excited, middle management skeptical
- Data teams overwhelmed with requests
- Governance lagging behind experimentation
- Vendors overselling, customers disappointed

## Questions I'm Wrestling With
- Is the current AI hype cycle different from previous ones?
- Why do enterprises struggle to move from POC to production?
- What separates successful AI adopters from the rest?

## Patterns I've Noticed
- Companies with strong data foundations do better
- Executive sponsorship necessary but not sufficient
- Starting with internal use cases reduces risk
- Change management often overlooked

## What I Used to Think
AI adoption is primarily a technology problem

## What I'm Starting to Think
It's fundamentally an organizational change problem

Step 2: Generate POV

forge generate pov --input ai-observations.md --mode thought-leadership --format html,word

Expected Output Characteristics

The generated POV includes:

  • Opening: Question-led hook (not thesis statement)
  • Body: Personal observations grounded in specific examples
  • Evolution: "I used to think... now I think..." progression
  • Tension: Self-interrogation without self-doubt
  • Conclusion: Provisional insights ("Here's where I've landed—for now")

Word Count: 800-1200 words

Output Files:

output/
├── pov_ai-observations_20241125_152330.html
└── pov_ai-observations_20241125_152330.docx

Use Case 8: Research-Enhanced Generation

What You'll Create: Content enriched with current web research.

Scenario: Write about a topic requiring current market context.

Two-Step Workflow

Step 1: Research Phase

forge agent research "enterprise AI governance frameworks 2024" \
  --max-results 15 \
  --output research.json

Step 2: Generation Phase

forge agent generate pov \
  --input observations.md \
  --research research.json \
  --format html

Single-Step Workflow

forge agent generate pov \
  --input observations.md \
  --research \
  --format html

Note: Single-step workflow automatically performs research based on input content.

Research Integration

The generated content will:

  • Cite specific sources inline
  • Include a references section
  • Ground claims in current data
  • Link to source materials

Example Citation:

According to the 2024 NIST AI Risk Framework, 80% of enterprises 
lack formal AI governance structures[1].

[1] NIST AI Risk Management Framework, 2024
    https://www.nist.gov/ai-framework

Tips by Use Case Category

Executive Content Best Practices

Input Preparation:

  • Include stakeholder names and roles
  • Specify budget and timeline constraints
  • Note political considerations
  • Clarify decision authority

Example:

# Context
Audience: Board of Directors (risk-averse)
Budget: $5M approved, $8M requested
Timeline: Must decide by Q1
Key concern: Regulatory compliance

Generation Tips:

# Specify advisory mode if auto-detect fails
forge generate brief --input exec-notes.md --mode advisory

# Include multiple formats for different uses
forge generate deck --input strategy.md --format pptx,pdf,html

Review Checklist:

  • Recommendations are specific and actionable
  • Budget and timeline are accurate
  • Risks are balanced (not overstated)
  • Decision request is clear

Technical Documentation Best Practices

Input Preparation:

  • Document specific technical decisions made
  • Include non-functional requirements
  • Note constraints (budget, timeline, skills)
  • Capture alternatives considered

Example:

# Technical Context
Current: Monolithic PHP app
Constraint: Must maintain uptime during migration
Team: 5 developers, limited Kubernetes experience
Timeline: 6 months to first production service

## Non-Functional Requirements
- 99.9% availability
- <200ms p95 latency
- GDPR compliant

Generation Tips:

# Use architecture mode for system docs
forge generate architecture --input design.md --mode architecture

# Generate ADRs for key decisions
forge generate adr --input decision.md --format markdown,html

Review Checklist:

  • Architecture diagrams are accurate
  • Technical decisions have rationale
  • Non-functional requirements documented
  • Security considerations included
  • Deployment approach specified

Thought Leadership Best Practices

Input Preparation:

  • Include your evolving thinking
  • Note questions you're wrestling with
  • Share what changed your mind
  • Ground observations in specific experiences

Example:

# My Evolving Thinking

## What I Used to Think
AI adoption is primarily about technology choices

## What Changed My Mind
- Worked with 5 enterprises on AI initiatives
- 4 of 5 failed due to organizational issues
- The one success had strong change management

## What I'm Starting to Think
The bottleneck is organizational, not technical

## Questions I'm Still Wrestling With
- How do you measure readiness?
- What's the right pace of adoption?

Generation Tips:

# Use thought-leadership mode
forge generate pov --input thinking.md --mode thought-leadership

# Add research for current context
forge agent generate pov --input thinking.md --research --format html

# Iterate for quality
forge agent generate pov --input thinking.md --iterate --format pdf

Review Checklist:

  • Opens with a question or tension
  • Shows evolution of thinking
  • Grounded in specific examples
  • Avoids prescriptive mandates
  • Includes provisional conclusions

Agentic Workflow Best Practices

When to Use Research:

# Use for topics requiring current data
forge agent generate pov --input analysis.md --research

# Use when you need citations
forge agent research "topic" --output research.json
forge agent generate paper --input paper.md --research research.json

When to Use Iteration:

# Use for critical content
forge agent generate brief --input critical.md --iterate

# Use when quality threshold is high
forge agent generate deck --input important.md --iterate --voice-threshold 8.0

# Use with custom iteration limits
forge agent generate pov --input article.md --iterate --max-iterations 5
