Signal Forge Use Cases Guide

Real-world examples showing how to transform rough notes into polished professional content.

Quick Start

Generate your first document in 60 seconds:

# 1. Create a simple input file
echo "# Meeting Notes
- Discussed cloud migration strategy
- Budget approved: $2M
- Timeline: 6 months
- Key risk: team capacity" > notes.md

# 2. Generate an executive brief
forge generate brief --input notes.md --format pdf

# 3. Find your output
ls output/briefs/

Core Concepts

Input Files

Signal Forge works with markdown files containing your raw content: meeting notes, observations, technical decisions, or research findings.

Document Types

  • Brief: Executive summaries and strategic documents
  • Deck: Presentation slide decks
  • Architecture: Technical architecture documentation
  • ADR: Architecture Decision Records
  • POV: Thought leadership and opinion pieces
  • Paper: Long-form research papers
  • Guide: User guides and documentation

Agentic Workflows

Add intelligence to generation with:

  • --research: Pull current web context
  • --iterate: Automatically refine until quality thresholds met
  • --memory: Maintain context across sessions

Use Case 1: Client Strategy Deck

What You'll Create: A polished 10-15 slide presentation from rough meeting notes.

Scenario: You've completed a strategy session and need a follow-up deck for stakeholders.

Step 1: Prepare Your Input

Create meeting-recap.md:

# Client Strategy Session - TechCorp
Date: November 25, 2024
Attendees: Sarah (CEO), Mike (CTO), Jennifer (Product)

## Key Discussion Points
- Current platform is aging, 7 years old
- Migration to cloud discussed but concerns about cost
- Customer complaints about slow performance increasing
- Competition launching new features quarterly

## Decisions Made
- Phased migration approach preferred
- Start with customer-facing systems
- Need ROI analysis before full commitment

## Open Questions
- Build vs buy for new commerce engine?
- Timeline expectations from board?

Step 2: Generate the Deck

forge generate deck --input meeting-recap.md --format pptx,html

Step 3: Review Output

Expected output structure:

Slide Content
1Executive Summary
2-3Current State Assessment
4-6Proposed Migration Approach
7-9Phased Timeline & Milestones
10-11Investment Framework & ROI
12-13Risk Considerations
14-15Recommended Next Steps

Use Case 2: Board-Ready Executive Brief

What You'll Create: A 2-page executive summary for board review.

Scenario: The CEO needs a concise brief summarizing a major initiative.

Input

# Digital Transformation Initiative Summary

## Initiative Overview
18-month program to modernize core systems

## Investment
$12M total - $4M in FY24, $8M in FY25

## Business Case
- 40% reduction in operational costs
- 3x faster time-to-market
- Improved customer satisfaction (NPS +20)

## Key Milestones
- Phase 1 (Q1-Q2): Foundation
- Phase 2 (Q3-Q4): Core systems
- Phase 3 (Q1-Q2 Y2): Integration

## Current Status
On track - 2 of 6 milestones complete

Generate

forge generate brief --input transformation-context.md --format word,pdf

Expected Output

Page 1: Executive Summary, Investment & ROI snapshot, Progress dashboard

Page 2: Key risks and mitigations, Critical dependencies, Board decision request

Use Case 3: Architecture Document

What You'll Create: Comprehensive architecture documentation with diagrams and decision rationale.

Scenario: Document a new microservices architecture after a design session.

Input

# Commerce Platform Redesign - Design Session

## Current Architecture
- Monolithic Java application
- PostgreSQL database
- Basic caching with Redis

## Proposed Architecture
- API Gateway for routing
- Separate services: Catalog, Cart, Checkout, Orders
- Event-driven communication via Kafka
- Each service owns its data

## Key Decisions
- Go with Kubernetes for orchestration
- Use GraphQL for frontend API
- PostgreSQL for transactional, MongoDB for catalog
- Auth via OAuth 2.0 + OIDC

## Non-functional Requirements
- 99.9% availability
- <200ms p95 latency
- Handle 10k concurrent users
- GDPR compliant

Generate

forge generate architecture --input design-session-notes.md --format html,pdf

Output Structure

Section Content
Executive SummaryHigh-level overview and key decisions
Context & ScopeBusiness drivers and system boundaries (C4 Level 1)
Container ArchitectureService descriptions and interactions (C4 Level 2)
Component DetailsInternal structure of each service
Data ArchitectureData flow and storage patterns
DeploymentInfrastructure and orchestration
SecurityAuthentication, authorization, data protection
ADRsRationale for key technical decisions

Use Case 4: Architecture Decision Record

What You'll Create: A structured ADR documenting a key technical decision.

Scenario: Document why you chose Kafka over alternatives for event streaming.

Input

# Event Streaming Decision Context

## The Decision
Need to choose event streaming platform for new architecture

## Options Considered
1. Apache Kafka
2. RabbitMQ
3. AWS SQS + SNS

## Evaluation Criteria
- Throughput: Need 100k msgs/sec
- Durability: Events must persist for replay
- Ordering: Strict ordering required per partition
- Team expertise: Some Kafka experience

## Outcome
Chose Kafka - best fit for throughput and durability requirements

Generate

forge generate adr --input kafka-decision.md --format html

Output Structure

ADR-style document: Title, Status, Context, Decision, Consequences

Use Case 5: Thought Leadership POV

What You'll Create: A 1000-word opinion piece for your blog or newsletter.

Scenario: Share observations about AI governance patterns you've noticed.

Input

# Observations on AI Governance

Been in conversations about AI governance lately.
Everyone's building frameworks, but nobody's using them.

Pattern I keep seeing:
- Company creates detailed AI governance policy
- Legal reviews it for 3 months
- Gets published on intranet
- Teams ignore it and use ChatGPT anyway

Why doesn't governance work?

Theory 1: Too rigid
Frameworks try to anticipate every scenario.
But AI moves too fast.

Theory 2: Wrong incentives
Governance is positioned as "what you can't do"
Teams see it as obstacle, not enabler

Generate

forge generate pov --input observations.md --format html,markdown

Voice Characteristics

  • Opens with observation, not thesis
  • Shows thinking evolution
  • Includes self-interrogation
  • Ends provisionally ("Here's where I've landed...")
  • Uses intentional fragments for rhythm

Use Case 6: Research-Backed Paper

What You'll Create: A 3000-5000 word paper with research citations.

Scenario: Write a thought leadership paper on AI governance trends.

Input

# AI Governance in Financial Services

Initial thoughts on how banks approach AI governance differently.

Observations:
- Regulatory pressure driving formal frameworks
- But frameworks often lag behind actual AI usage
- Gap between policy and practice

Questions to explore:
- What makes some governance frameworks stick?
- How do leading institutions balance innovation and control?

Generate with Research

forge agent generate paper \
  --input ai-governance-notes.md \
  --research \
  --iterate \
  --format pdf,html

Workflow

  1. Research agent searches for current context
  2. Draft generated with citations
  3. Voice check performed
  4. Iteration until score reaches 7+

Use Case 7: Research-Only Workflow

What You'll Create: Research context to feed into future generation.

Scenario: Gather information before writing a comprehensive paper.

Run Research

forge agent research "AI governance frameworks 2024" \
  --output research-context.json

Use Research in Generation

forge agent generate paper \
  --input observations.md \
  --research-context research-context.json \
  --format pdf

Use Case 8: Iterative Refinement

What You'll Create: Auto-refined content that meets voice quality thresholds.

Scenario: Ensure your executive deck meets quality standards.

Generate with Iteration

forge agent generate deck \
  --input client-notes.md \
  --iterate \
  --max-iterations 5 \
  --format pptx

Expected Workflow

✓ Draft generated
✓ Voice check: 5.2/10
  Issues: Opening too tentative, lacks business context
✓ Revision 1: 6.8/10
  Issues: Recommendations need more confidence
✓ Revision 2: 8.1/10 - approved
✓ Generated: output/deck-2024-01-15.pptx
