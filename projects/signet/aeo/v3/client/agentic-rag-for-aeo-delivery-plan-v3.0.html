Profound Setup Requirements

Overview

Profound is the data source that powers the measurement system. If this data is incomplete or incorrect, the system produces nothing of value. Three specific artifacts must be delivered before development begins.

About Profound: Profound is an enterprise AEO (Answer Engine Optimization) platform that tracks brand visibility across AI answer engines. The platform provides citation tracking, sentiment analysis, and share of voice metrics via REST API.

Artifact A: The Prompt Bank

Property Details
Owner Content Strategist (with AEO Strategist review)
Format CSV or Excel
Volume 400 prompts minimum (pilot scope)
Due Date Week 0 (before development starts)

What it is: A curated list of questions users ask AI answer engines, categorized by intent.

Required Columns

Column Description Example
prompt_id Unique identifier PRD-001
query_text The natural language prompt "Best running shoes for flat feet"
intent_category Primary user intent purchase, research, compare
sub_category Product/topic grouping footwear, accessories
priority Business importance (1-5) 5

Example Prompts by Intent

Intent Example Prompt
Purchase "Best [product] under $500"
Research "[Product category] trends 2025"
Compare "[Brand A] vs [Brand B] quality"
Trust "Is [brand] reliable?"
How-to "How to choose [product] for [use case]"

Why needed: Profound executes scans against these prompts across all configured answer engines. The prompts you track determine what visibility data you receive.

Artifact B: Competitor List

Owner: Client Stakeholder + AEO Strategist | Due: Week 0

Provide 3-5 competitor domain names (e.g., competitor-one.com, competitor-two.com).

Why needed: Profound calculates Share of Voice by comparing your citation frequency against these competitors. Without this list, SOV metrics cannot be computed.

Artifact C: API Access & Configuration

Property Details
Owner Client IT + AEO Strategist
Format API key and account confirmation

Requirements

Why needed: Without API access, we cannot retrieve visibility data. The Profound account must be configured with prompts and competitors before scans produce useful data.

PIM Integration - Approach 1 (Pre-Indexed SKU Catalog)

Overview

Ingest the client's full product catalog into OpenSearch alongside Profound data. Use semantic vector matching to identify which SKUs correspond to visibility gaps.

How It Works

Step Action System
1 Client exports product catalog (SKU, title, description, attributes) PIM → S3
2 Catalog Ingestion Lambda embeds product data into vectors AWS Lambda + Titan
3 SKU vectors indexed in OpenSearch alongside Profound data OpenSearch
4 Profound identifies gap: "women's gold watches" visibility low Profound → AWS
5 System embeds "women's gold watches" and runs similarity search Bedrock + OpenSearch
6 Returns matching SKUs: SKU-1234, SKU-5678 (similarity > 0.85) OpenSearch
7 Agent generates optimized content for each matched SKU Bedrock Agent
8 PIM Update Lambda pushes content to Infoverity via API Lambda → PIM API

Trade-offs

Pros Cons
Precise semantic matching Requires catalog sync to AWS
Works with sparse Profound tags 100K SKUs = storage + embedding cost
Handles edge cases (gold vs silver) Sync lag = potentially stale matches
Fully autonomous Another system with product data
Tunable similarity threshold False positives if products semantically close

Non-Negotiables

  • PIM must provide catalog export capability
  • PIM must have write API for automated updates
  • Daily sync frequency minimum

PIM Integration - Approach 2 (Tag-Based Mapping)

Overview

Client maintains a mapping table that links Profound topics/tags to PIM category IDs. System uses this mapping to identify which SKUs need content updates—no semantic inference required.

How It Works

Step Action System
1 Client creates mapping: Profound Tag → PIM Category ID Manual (spreadsheet)
2 Mapping table loaded into DynamoDB AWS Lambda
3 Profound identifies gap: "women's gold watches" visibility low Profound → AWS
4 System looks up tag in mapping table DynamoDB
5 Returns mapped PIM category: WATCHES-WOMENS-GOLD DynamoDB
6 System queries Infoverity API for SKUs in that category Lambda → PIM API
7 Agent generates optimized content for returned SKUs Bedrock Agent
8 PIM Update Lambda pushes content to Infoverity via API Lambda → PIM API

Trade-offs

Pros Cons
Deterministic—no false positives Requires manual mapping maintenance
No SKU data in AWS New tags require client action
Simple to understand and debug Coverage limited to mapped tags
Client controls the mapping Doesn't handle edge cases automatically
Lowest infrastructure cost Scales linearly with tag count

Non-Negotiables

  • Client must commit to maintaining mapping table
  • Profound tag taxonomy must be documented
  • PIM must have read API for category-based SKU queries
  • PIM must have write API for automated updates

PIM Integration - Approach 3 (Real-Time Embedding)

Overview

Query Infoverity at runtime to retrieve relevant SKUs, embed them on-the-fly, and perform semantic matching against Profound gaps. No pre-indexed catalog—embeddings generated per request.

How It Works

Step Action System
1 Profound identifies gap: "women's gold watches" visibility low Profound → AWS
2 System extracts category hint from Profound data (e.g., "watches") Bedrock
3 Query Infoverity API for SKUs in related categories Lambda → PIM API
4 PIM returns candidate SKUs (50-500 per query) Infoverity
5 Embed candidate SKU data in real-time Bedrock Titan
6 Embed Profound gap query Bedrock Titan
7 Compute similarity, select matches above threshold Lambda
8 Agent generates optimized content for matched SKUs Bedrock Agent
9 PIM Update Lambda pushes content to Infoverity via API Lambda → PIM API

Trade-offs

Pros Cons
No SKU data stored in AWS Higher latency per request (2-5 sec)
Always uses current PIM data Embedding cost scales with query volume
Handles new SKUs immediately Requires category inference (error-prone)
Flexible—adapts to any taxonomy PIM API rate limits may throttle

Non-Negotiables

  • PIM must have read API for category-based SKU queries
  • PIM API must support query volumes without heavy rate limiting
  • PIM must have write API for automated updates
  • Acceptable latency budget: 5+ seconds per gap processing

PIM Integration - Approach 4 (Hybrid - Recommended)

Overview

Combine tag-based mapping with semantic fallback. Use deterministic mappings where they exist; fall back to real-time semantic matching for unmapped or ambiguous cases. Best of both worlds for POC flexibility.

How It Works

Step Action System
1 Profound identifies gap: "women's gold watches" visibility low Profound → AWS
2 Check mapping table for Profound tag DynamoDB
3a If mapped: Query PIM for SKUs in mapped category Lambda → PIM API
3b If unmapped: Extract category hint, query PIM broadly Bedrock + PIM API
4 For unmapped: Embed candidates and gap, compute similarity Titan + Lambda
5 Agent generates optimized content for matched SKUs Bedrock Agent
6 PIM Update Lambda pushes content to Infoverity via API Lambda → PIM API
7 Log unmapped tags for client review CloudWatch + Report

Why Recommended for POC

Factor Benefit
Reduced upfront effort Start with 20-30 high-value mappings, not 200
Demonstrates both paths Shows deterministic and semantic capabilities
Data-driven expansion Unmapped tag logs inform Phase 2 scope
Lower risk Fallback prevents complete failure on unmapped tags
Client engagement Mapping review creates natural touchpoint

Trade-offs

Pros Cons
Deterministic where possible More complex system
Graceful handling of unmapped tags Two code paths to maintain
Client can expand coverage over time Higher initial build effort
Self-documenting coverage gaps Semantic fallback still has false positives

Non-Negotiables

  • Client commitment to mapping table maintenance
  • Weekly review of unmapped tag reports
  • PIM read/write API access
  • Initial mapping set delivered Week 1

PIM Integration - Approach Comparison

Executive Summary

Four approaches to bridge Profound query-level insights to PIM SKU-level actions. Recommendation: Approach 4 (Hybrid) for POC—balances automation with pragmatic client involvement.

Comparison Matrix

Criterion Approach 1 Approach 2 Approach 3 Approach 4
Pre-Index SKUs Tag Mapping Real-Time Embed Hybrid
Accuracy High Deterministic Medium-High High
Maintenance Sync jobs Client mapping None Light mapping
SKU Data in AWS Yes No No No
Handles New Tags Auto Manual Auto Auto (fallback)
Build Complexity Medium Low Medium-High Medium-High
Runtime Cost Low Lowest Medium Low-Medium
Latency Fast Fastest Slow (2-5s) Fast/Medium
POC Suitability Medium Medium Low High

Decision Factors

Choose Approach 1 (Pre-Index) if:

  • Client comfortable with SKU data in AWS
  • 100K+ SKUs with rich metadata
  • Tags are sparse or poorly aligned with categories

Choose Approach 2 (Tag Mapping) if:

  • Tags align well with PIM categories
  • Client has bandwidth for mapping maintenance
  • Simplicity and determinism valued

Choose Approach 3 (Real-Time) if:

  • SKU catalog changes frequently
  • No pre-work possible
  • Latency not critical

Choose Approach 4 (Hybrid) if:

  • POC scope with expansion planned
  • Unknown tag-to-category alignment
  • Need graceful degradation

Architecture Review Questions

  1. Data residency: Is SKU data in AWS acceptable?
  2. Mapping ownership: Who maintains tag-to-category mappings?
  3. Latency tolerance: Real-time (5s) vs batch processing acceptable?
  4. PIM API capabilities: Confirm read/write endpoints and rate limits
  5. Taxonomy alignment: How well do Profound tags match PIM categories?
