Section 1: Solution Overview

The Problem

Large Language Models (ChatGPT, Perplexity, Google AI Overviews) increasingly influence purchase decisions. When consumers ask AI assistants about jewelry products, Signet (Kay Jewelers, Jared) has:

  • No visibility into how they're represented in LLM responses
  • No tools to improve that representation
  • No measurement of before/after optimization impact

The Solution (30-Second Pitch)

An AWS-native agentic RAG system that:

  1. Ingests brand visibility data from Profound API (daily LLM scans)
  2. Stores data in OpenSearch vector database for semantic search
  3. Analyzes visibility gaps using Bedrock Agent (Claude)
  4. Recommends content optimizations to close gaps
  5. Measures Share of Voice (SOV) lift via weekly delta calculations
  6. Exports recommendations to Infoverity PIM for publishing

Expected Outcomes (Pilot)

Metric Target Measurement Window
Visibility gap closure 60% of identified gaps 90 days post-pilot
Share of Voice lift +5% aggregate Pilot end (Week 12)
Content optimizations 100+ per month Ongoing post-pilot
System availability 99.5% uptime Continuous
Query performance <2s response (P95) Continuous

Architecture Pattern: Event-Driven Agentic RAG

Profound API → S3 (raw data lake) → Lambda (ingestion)
    → OpenSearch (vector store) → Bedrock Agent (orchestration)
    → Tools (Query, Analyze, Recommend, Measure) → User/PIM

Key Innovation: Instead of hardcoded workflows, a Bedrock Agent learns to route user requests to the appropriate tool based on natural language intent.

Data Flows

Flow 1: Data Ingestion (Daily, Automated)

1. Profound API → S3 bucket (profound-exports/)
2. S3 PUT event → Ingestion Lambda trigger
3. Lambda validates JSON schema
4. Lambda calls Bedrock Titan for embeddings
5. Lambda writes document + vector to OpenSearch
6. CloudWatch logs success/failure

Flow 2: User Query (Ad-hoc)

1. User → API Gateway (authenticated)
2. API Gateway → Request Handler Lambda
3. Request Handler → Bedrock Agent
4. Agent classifies intent (Query vs Analyze vs Measure)
5. Agent invokes appropriate tool Lambda
6. Tool queries OpenSearch (kNN + filters)
7. Tool returns context to Agent
8. Agent generates grounded response
9. Response → User (with citations)

Flow 3: Weekly Measurement (Scheduled)

1. EventBridge (Sunday 2am UTC) → Distillation Lambda
2. Lambda queries OpenSearch for T0 and T1 data
3. Lambda calculates SOV deltas (per-query and aggregate)
4. Lambda writes compressed report to S3 (aeo-reports/weekly/)
5. CloudWatch logs completion

Key Architecture Decisions

ADR-001: Agentic Intent Routing

Decision: Use Bedrock Agent for orchestration instead of rule-based routing.

Context: User queries are ambiguous. "Why is visibility dropping on gold chains and what should I do?" requires:

  1. Semantic search for "gold chains" data
  2. Delta analysis vs previous period
  3. Recommendation generation

Traditional routing would require complex conditional logic and constant updates.

Consequences:

  • ✅ Natural language understanding handles ambiguity
  • ✅ New tools can be added via configuration (no code changes)
  • ✅ Multi-step queries handled automatically
  • ❌ Added latency (~200-400ms for agent reasoning)
  • ❌ Less predictable behavior vs deterministic routing

Trade-off Accepted: Flexibility > Latency (2s target still achievable)

ADR-002: Pre-Calculated Measurement

Decision: Run weekly distillation jobs instead of real-time delta queries.

Context: Measurement queries ("Did last week's changes help?") require comparing T0 vs T1 data across 400 prompts. Real-time calculation would:

  • Take 5-10 seconds per query
  • Cost $0.50+ per query (Bedrock + OpenSearch compute)
  • Return inconsistent results (different time windows per user)

Consequences:

  • ✅ Instant response (<100ms to retrieve pre-calculated report)
  • ✅ Consistent metrics across all users
  • ✅ 95% cost reduction vs real-time
  • ❌ Data can be up to 7 days stale
  • ❌ Cannot answer ad-hoc time range queries

Trade-off Accepted: Performance + Cost > Freshness (weekly cadence aligns with content optimization cycles)

ADR-003: Grounding Enforcement

Decision: System prompt enforces retrieval-before-generation pattern.

Context: LLMs can hallucinate plausible-sounding insights not grounded in actual Profound data. This creates:

  • Trust issues with stakeholders
  • Business risk (acting on false insights)
  • Audit failures (no source data trail)

Implementation: System prompt includes:

"You MUST query the knowledge base before generating any response. NEVER generate content without retrieved context. Include confidence level based on retrieval quality. If retrieval returns no results, respond with 'I don't have data on that query—please check if it's in the Profound scan scope.'"

Consequences:

  • ✅ All responses tied to source data
  • ✅ Auditable citations in every response
  • ✅ Stakeholder trust (can verify claims)
  • ❌ Cannot answer questions outside Profound scope
  • ❌ Longer prompts increase token costs (~10%)

Trade-off Accepted: Accuracy > Flexibility (hallucination is unacceptable for business decisions)

ADR-004: No Low-Code Orchestration

Decision: Use native AWS services (Lambda, EventBridge) instead of n8n, Zapier, or similar.

Context: Low-code platforms could accelerate initial development but introduce:

  • Additional platform to secure, monitor, patch
  • Middleware latency (200-500ms per hop)
  • Vendor lock-in and licensing costs
  • Reduced control over error handling

Consequences:

  • ✅ Single platform to manage (AWS only)
  • ✅ Better performance (no middleware)
  • ✅ Cost efficiency (pure serverless billing)
  • ✅ Full control over retry logic, error handling
  • ❌ Requires stronger AWS/Python skills
  • ❌ No visual workflow designer for non-engineers

Trade-off Accepted: This is a "Special Forces" approach—smaller skilled team > larger general team.

    Stack,
    aws_s3 as s3,
    RemovalPolicy,
    Duration

class StorageStack(Stack):
    def __init__(self, scope, id, **kwargs):
        super().__init__(scope, id, **kwargs)

            self, "RawExports",
            bucket_name="aeo-profound-exports",
            versioned=True,
            lifecycle_rules=[
                s3.LifecycleRule(

        # Compressed reports
        self.reports_bucket = s3.Bucket(
            self, "Reports",
            bucket_name="aeo-reports",
            versioned=True,
            removal_policy=RemovalPolicy.RETAIN

Section 6: Decision Framework

How to Make Trade-Offs

During Plan/Analyze/Design, you'll encounter ambiguous requirements and conflicting priorities. Use this framework:

Decision Tree

1. Does this decision impact the Critical Path (Phases 0→1→2→5)?
   YES → Escalate to DL/PM + Client Stakeholder
   NO → Proceed to step 2

2. Does this decision violate an ADR or architectural principle?
   YES → Document as ADR amendment; review with team
   NO → Proceed to step 3

3. Does this decision require client input (preference, not technical)?
   YES → Add to weekly status agenda; document in Decision Log
   NO → Proceed to step 4

4. Is this decision reversible (can be changed later without major rework)?
   YES → Make decision; document rationale; move on
   NO → Escalate to Lead Architect for review

Example Decisions (Week 1-2)

Decision Apply Framework Outcome
"Should we use Claude 3 Sonnet or Claude 3.5 Sonnet?" Step 2: Not an ADR violation. Step 4: Easily reversible (model parameter). Lead Architect decides: Use 3 Sonnet (SOW pricing based on this). Upgrade to 3.5 if needed in Week 5.
"Should we support real-time queries or daily batch only?" Step 2: Violates ADR-002 (pre-calculated reporting). Step 2: Requires ADR amendment. Escalate to team + DL/PM: Real-time = major architecture change. Confirm with client if required. If yes, revise SOW.
"Should we prioritize rings or necklaces for pilot?" Step 3: Client preference, not technical. Add to kickoff agenda: Let client decide based on business priority.
"Should Lambda timeout be 5 minutes or 15 minutes?" Step 4: Easily reversible (config change). Lead Architect decides: Start with 5 min; increase if needed during testing.

When to Escalate

Escalate to DL/PM:

  • Timeline impact (any delay to critical path)
  • Budget impact (>10% cost increase)
  • Scope creep (client requests new features)
  • Stakeholder misalignment (Signet teams disagree)

Escalate to Client:

  • Business priority decisions (which product category?)
  • Security policy questions (data retention, PII handling)
  • Integration ownership disputes (ESI vs Infoverity responsibility)

Escalate to Outgoing Architect (Week 0-1 only):

  • Fundamental assumption invalidated (e.g., Profound schema is completely different)
  • ADR reconsideration (new information changes trade-off)

Section 8: Technical Deep Dives

Deep Dive 1: Bedrock Agent Configuration

What Is a Bedrock Agent?

A Bedrock Agent is an orchestration layer that:

  1. Takes natural language input from user
  2. Classifies intent (what is the user trying to do?)
  3. Selects the appropriate tool(s) to invoke
  4. Calls tool Lambda functions in sequence (if needed)
  5. Synthesizes tool outputs into a coherent response
  6. Returns response with citations

Description: Answer Engine Optimization assistant for Signet

Deep Dive 2: OpenSearch k-NN Search

How Semantic Search Works:

  1. Indexing (Ingestion Lambda):

Why k-NN?:

  • Semantic similarity: "engagement rings" matches "wedding bands", "promise rings"
  • Handles synonyms: "gold chain" matches "gold necklace"
  • No keyword brittleness: "best jewelry for men" finds "men's accessories"

Performance: <100ms at 250K documents (with ef_search: 512)

Deep Dive 3: Grounding Enforcement

The Hallucination Problem:

Without grounding, an LLM might say:

"Visibility is dropping because your descriptions lack emotional appeal. Competitors use phrases like 'timeless elegance' which resonate with buyers."

Problem: This sounds plausible but might be completely made up if Profound data doesn't actually show competitors using those phrases.

The Solution: Forced Retrieval:

System prompt enforces a two-step process:

  1. Step 1: Retrieve (mandatory)

    • Agent MUST call query_tool or analyze_tool
    • Tools return actual Profound data with citations

  2. Step 2: Generate (only with context)

    • Agent uses retrieved data to formulate response
    • Every claim must reference a source query or data point

Implementation:

System Prompt (excerpt):
  "Before answering any question, you MUST:
  1. Use query_tool to search the knowledge base
  2. If query_tool returns 0 results, respond: 'I don't have data on that query'
  3. If query_tool returns results, use ONLY that data to answer
  4. Include citations in format: [Source: Profound scan YYYY-MM-DD, query 'X']
  5. Never speculate or infer beyond what the data explicitly shows"

Example Enforced Response:

"Visibility on 'engagement rings' has Share of Voice (SOV) 0.64, ranking 3rd behind BlueNile (0.78) and JamesAllen (0.71).

Gaps identified:

  • 15 queries mention 'lab-grown diamonds'; Kay appears in 0 responses
  • Competitors cite 'conflict-free certification' in 23% of responses; Kay in 8%

[Source: Profound scans 2024-10-01 to 2024-11-01, 87 queries analyzed] Confidence: 91%"

Validation: In Week 5 testing, reject any response that lacks citations.

Section 9: Common Pitfalls & How to Avoid

Pitfall 1: Starting Development Before Requirements Lock

Symptom: Lead Architect writes Lambda code in Week 1

Why Bad: Profound schema might change; AWS account might not be ready; IAM policies undefined

Fix: Week 1-2 are design only. Write CDK infrastructure skeleton (no application logic). Actual Lambda development starts Week 3.

Pitfall 2: Assuming Signet Teams Are Aligned

Symptom: Assuming ESI Team knows about this project

Why Bad: ESI might be unaware; Infoverity might have conflicting priorities; integration gets delayed

Fix: Direct outreach to ESI and Infoverity Week 1 Day 2. Don't rely on Signet PM alone to cascade information.

Pitfall 3: Over-Engineering for Scale

Symptom: Designing for 10M documents, real-time streaming, multi-region failover

Why Bad: This is a POC with 250K documents. Complexity kills delivery speed.

Fix: Right-size for pilot. Use ADRs to document "what we're NOT doing" (e.g., "No multi-region for POC; document as Phase 2 consideration").

Pitfall 4: Ignoring the Critical Path

Symptom: Spending time on nice-to-have features (dashboards, advanced monitoring)

Why Bad: Critical path is Phase 0 → 1 → 2 → 5. Anything else is distraction.

Fix: Use the dependency diagram (phase-dependencies.excalidraw). If a task isn't on the critical path, defer it.

Pitfall 5: Not Enforcing Grounding from Day 1

Symptom: Testing agent responses without checking for citations

Why Bad: Hallucinations will slip through; discovered late in pilot; trust is lost

Fix: Add "citation check" to every test case. Reject any response without [Source: ...] attribution.
