Section 3: Architecture Knowledge Transfer

System Context (30,000-Foot View)

What This System Does:

  • Ingests brand visibility data from Profound API (daily scans of LLM responses)
  • Stores data in OpenSearch vector database for semantic search
  • Uses Bedrock Agent (Claude) to analyze gaps and generate optimization recommendations
  • Measures before/after visibility lift via weekly SOV delta calculations
  • Exports recommendations to Infoverity for PIM updates

Why This Architecture:

  • Event-driven: Profound data arrives asynchronously; EventBridge triggers processing
  • Serverless: No servers to manage; auto-scaling; pay-per-use
  • Agentic: Bedrock Agent handles ambiguous queries requiring multi-step reasoning
  • Pre-calculated: Weekly measurement reports for instant "Did it work?" answers

Key Architecture Decisions (ADRs)

Decision Choice Trade-off Rationale
ADR-001: Intent Routing Bedrock Agent (not rule-based) +Flexibility, -Latency (~300ms) User queries are ambiguous; need NLU
ADR-002: Measurement Pre-calculated weekly +Performance, -Freshness (7 days) Instant answers vs real-time accuracy
ADR-003: Grounding Enforced via system prompt +Accuracy, -Flexibility Prevent hallucination in recommendations
ADR-004: Orchestration Native AWS (no n8n) +Performance, -Requires AWS skills Smaller skilled team > larger general team

Critical Insight: The "agentic" pattern is the core innovation. Instead of writing conditional logic for every query type, the Bedrock Agent learns to route requests to the right tool (Query, Analyze, Recommend, Measure) based on user intent.

Data Flows to Understand

Flow 1: Ingestion (Daily)

Profound API → S3 (raw exports) → Lambda (triggered by S3 event)
    → Titan Embeddings → OpenSearch (vector + metadata)

Flow 2: Query (Ad-hoc)

User → API Gateway → Request Handler Lambda → Bedrock Agent
    → Tool Selection (Query/Analyze/Recommend) → OpenSearch → Bedrock Generation → User

Flow 3: Measurement (Weekly)

EventBridge (Sunday 2am) → Distillation Lambda
    → OpenSearch (T0 vs T1 comparison) → S3 (compressed report)

Integration Boundaries

Boundary Owner Accenture Responsibility Signet Responsibility
Inbound ESI Team Define API schema for S3 delivery Implement Profound → S3 integration
Outbound Infoverity Define recommendation output schema Consume API and update PIM/P360
Infrastructure Signet IT Implement AWS resources Provision accounts, approve ARB

Key Point: Accenture designs the integration specifications (OpenAPI schemas). Signet internal teams (ESI, Infoverity) implement their respective integrations.

Section 4: Technology Stack Deep Dive

AWS Services Used

Service Purpose Configuration Notes
S3 Raw data lake, compressed reports Lifecycle: Standard → Glacier after 90 days
Lambda Ingestion, request handling, distillation Python 3.11, 1-4GB memory, 5-15min timeouts
OpenSearch Vector search (kNN), metadata filtering 3-node cluster, 1024-dim Titan embeddings
Bedrock Agent orchestration (Claude), embeddings (Titan) Claude 3 Sonnet, token budgets enforced
EventBridge Scheduling (weekly distillation) Sunday 2am UTC cron trigger
API Gateway Public REST endpoint API Key + IAM auth
IAM Service-to-service auth Least-privilege roles (see Architecture v2 Slide 15)

OpenSearch Index Schema

The vector database stores Profound scan results with this structure:

{
  "scan_id": "prf_20241115_001",
  "brand_name": "Kay Jewelers",
  "query_text": "best gold chain for men",
  "query_vector": [0.234, -0.567, ...],  // 1024-dim Titan embedding
  "share_of_voice": 0.72,
  "sentiment_score": 0.85,
  "response_rank": 2,
  "timestamp": "2024-11-15T08:00:00Z",
  "llm_source": "chatgpt",
  "product_category": "chains",
  "competitor_mentions": [
    {"domain": "jared.com", "rank": 1, "sentiment": "positive"},
    {"domain": "zales.com", "rank": 3, "sentiment": "neutral"}
  ],
  "citations": ["kay.com", "jared.com"]
}

Key Fields for Search:

  • query_vector: Semantic similarity search (kNN)
  • product_category: Filter to pilot scope
  • share_of_voice: Measurement calculations
  • timestamp: Time-series analysis

Bedrock Agent Configuration

The agent is configured with 3 action groups (tools):

Tool Lambda Purpose Input Schema
Query Tool aeo-query-lambda Search knowledge base for visibility data {query_text, filters}
Analyze Tool aeo-analyze-lambda Identify gaps, generate recommendations {query_text, category}
Measure Tool aeo-measure-lambda Retrieve pre-calculated SOV deltas {time_range}

System Prompt (enforces grounding):

"You MUST query the knowledge base before generating any response. NEVER generate content without retrieved context. Include confidence level based on retrieval quality. If retrieval returns no results, respond with 'I don't have data on that query—please check if it's in the Profound scan scope.'"

Section 5: Critical Success Factors

For the Incoming Lead Architect

Week 1 Priorities:

  1. Validate Profound API access and schema (Architecture v2 Slide 4, Artifact C)
  2. Confirm AWS account provisioning and IAM permissions
  3. Design inbound API specification for ESI Team
  4. Review OpenSearch cluster sizing with Signet IT

Skills to Leverage:

  • OpenSearch k-NN configuration (critical for semantic search)
  • AWS CDK (all infrastructure as code)
  • IAM role design (least-privilege model)
  • Python/Boto3 for Lambda development

Pitfalls to Avoid:

  • Don't start coding without Profound schema validation (risk: schema change breaks ingestion)
  • Don't over-architect for scale (this is a POC; right-size for pilot volume)
  • Don't skip the IAM design review with Signet IT (security approval required)

For the Incoming AI Engineer

Week 3-4 Priorities (when you join):

  1. Configure Bedrock Agent with 3 action groups
  2. Write system prompt enforcing grounding
  3. Implement tool invocation logic in Lambda
  4. Test intent routing accuracy

Skills to Leverage:

  • Bedrock Agent configuration (action groups, knowledge base integration)
  • Prompt engineering (system prompts, few-shot examples)
  • Lambda development (Python)
  • JSON schema design for tool inputs/outputs

Pitfalls to Avoid:

  • Don't assume the agent will "just work" without tuning (expect 3-5 iterations on prompts)
  • Don't hardcode tool logic (use Bedrock's orchestration; let the agent decide)
  • Don't skip grounding enforcement (business risk if recommendations are hallucinated)

For the Incoming DL/PM

Week 1 Priorities:

  1. Establish governance cadence (weekly status, bi-weekly executive)
  2. Confirm Signet PM assignment (critical dependency)
  3. Validate Signet strategist availability (AEO Strategist, Content Strategist)
  4. Set up collaboration tools (Slack, Confluence, Jira)

Skills to Leverage:

  • Multi-stakeholder coordination (Accenture + Signet IT + ESI + Infoverity + ecom)
  • Risk management (proactive mitigation, not reactive firefighting)
  • Agile delivery (2-week sprints aligned to phases)

Pitfalls to Avoid:

  • Don't assume Signet teams are aligned (ESI and Infoverity may not know about this project)
  • Don't skip the Signet PM assignment (without it, coordination collapses)
  • Don't wait for blockers to resolve themselves (escalate early, escalate often)

Section 11: Post-Handoff Support

Availability Window

Outgoing Architect: Available for questions via Slack/Email during:

  • Week 0 (Kickoff Week): Full availability (respond within 2 hours)
  • Week 1: Best-effort (respond within 24 hours)
  • Week 2+: Emergency escalation only (critical blockers)

Preferred Contact Methods

  1. Slack (fastest): Direct message for quick clarifications
  2. Email (async): For detailed questions requiring documentation review
  3. Zoom (last resort): 30-min sync if written communication insufficient

Escalation Criteria

Contact outgoing architect immediately if:

  • Fundamental architecture assumptions are invalidated (e.g., Profound schema is completely different)
  • Client requests scope change that conflicts with ADRs
  • Critical dependency blocker with no obvious mitigation
  • Security review rejects IAM design

Do NOT contact for:

  • Normal implementation questions (trust the team's expertise)
  • Client preference questions (DL/PM should handle)
  • Routine project management issues
