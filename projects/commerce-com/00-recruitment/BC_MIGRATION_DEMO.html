BC Migration Platform

WooCommerce → BigCommerce Migration • December 2025 • Web Dashboard + CLI WP Engine Partnership

What This Platform Does

BC Migration is a comprehensive migration platform built specifically for the BigCommerce and WP Engine partnership. It combines a Next.js web application (merchant-facing assessment dashboard) with a CLI tool (automated data migration engine) to provide end-to-end migration capabilities.

The Web Dashboard: Merchants connect their WooCommerce store, receive instant migration readiness assessment, view detailed compatibility reports, and get personalized path recommendations (Path 1, 2, or 3).

The CLI Tool: Solutions Architects or agency partners use command-line tools to execute actual data migrations (products, categories, customers, orders) with checkpoint resume, validation, and error handling.

This complements the Catalyst-WordPress integration (Path 2 architecture). Where the WordPress content component handles content migration (blogs, pages), BC Migration handles data migration (products, orders, customers).

Together, these tools provide complete WooCommerce → BigCommerce migration capabilities with both merchant self-service (web UI) and professional services (CLI) options.

Core Migration Capabilities

Pre-Migration Assessment
Analyze WooCommerce store for migration readiness. Generates detailed reports on data volume, complexity, and potential blockers before any data is moved.
Idempotent Operations
Migrations can be safely re-run without creating duplicate records. Failed migrations can resume from last checkpoint without data corruption.
Delta Migrations
Only migrate items modified since last run. Enables incremental migrations for large stores or ongoing sync during transition periods.
Referential Integrity
Follows proper migration sequence (categories → products → customers → orders) to maintain data relationships and prevent orphaned records.
Rate Limiting & Backoff
Respects BigCommerce API limits (150 requests / 30 seconds). Implements exponential backoff for failed requests and batch processing for large datasets.
Post-Migration Validation
Verifies data completeness by comparing record counts and sampling price accuracy between WooCommerce source and BigCommerce destination.

Migration Workflow

Step 1: Assessment

Command: npm run assess

What It Does: Analyzes WooCommerce store structure, counts products/categories/customers/orders, identifies potential migration blockers (custom fields, unsupported plugins, data quality issues).

Output: Assessment dashboard showing readiness score, estimated migration time, and recommended migration strategy (Path 1, 2, or 3).

Step 2: Category Migration

Command: npm run migrate:categories

What It Does: Migrates WooCommerce product categories to BigCommerce, maintaining hierarchy and parent-child relationships.

Validation: Compares category count and structure between source and destination.

Step 3: Product Migration

Command: npm run migrate:products

What It Does: Migrates products with variants, images, pricing, inventory levels, and category assignments. Handles complex product types (simple, variable, grouped).

Validation: Samples 10% of products to verify price accuracy, image URLs, and variant configurations.

Step 4: Customer Migration

Command: npm run migrate:customers

What It Does: Migrates customer records including addresses, customer groups, and account metadata.

Security Note: Does NOT migrate passwords (customers must reset). Maintains GDPR compliance.

Step 5: Order History Migration

Command: npm run migrate:orders

What It Does: Migrates historical orders with line items, shipping info, payment status, and customer associations.

Validation: Verifies order totals, item counts, and customer references.

Step 6: Validation & Reconciliation

Command: npm run validate

What It Does: Comprehensive data validation comparing WooCommerce source vs. BigCommerce destination. Generates reconciliation report highlighting discrepancies.

Output: Pass/fail migration report with detailed logs for troubleshooting any issues.

Real-Time Assessment Engine

How It Works:

  1. Merchant enters WooCommerce credentials in web form
  2. Next.js API routes proxy requests to WooCommerce REST API
  3. Assessment collectors fetch data (products, categories, etc.) with pagination
  4. Client-side React components display progress bars as data streams in
  5. Results cached in localStorage for instant re-display on page refresh
  6. Assessment complete in 30-60 seconds for typical stores (<5,000 products)
Merchant Self-Service UX

Problem Solved: Traditional migration assessments require BigCommerce SA or WP Engine agency partner to manually audit WooCommerce store. Takes 2-3 days, costs merchant time and money.

BC Migration Dashboard Solution: Merchants get instant self-service assessment in <60 seconds. No SA required for initial scoping. Enables merchant self-qualification before engaging professional services.

Impact: Reduces sales cycle friction. Merchants see personalized migration path and timeline estimate before talking to sales. Improves conversion because merchants understand what they're buying.

Key Design Patterns

Pattern 1: Checkpoint Resume

Migration state is persisted to SQLite after each batch. If migration fails (network error, API timeout, rate limit), restart from last successful checkpoint rather than beginning.

Benefit: Large store migrations (10,000+ products) can survive interruptions without data loss or duplication.

Pattern 2: Batch Processing with Rate Limiting

Products/orders processed in batches of 50-100 items. Each batch respects BigCommerce rate limits (150 req/30s) with exponential backoff on 429 responses.

Benefit: Maximizes throughput while preventing API throttling or account suspension.

Pattern 3: Dry-Run Mode

All migration commands support --dry-run flag to simulate migration without writing data. Shows exactly what would happen without risk.

Benefit: Merchants can preview migration impact before committing to data changes.

Integration with Catalyst-WordPress

BC Migration and Catalyst-WordPress tools work together to provide complete migration coverage:

Migration Component Tool What It Migrates
Commerce Data BC Migration CLI Products, categories, customers, orders, inventory
Content Catalyst-WordPress Component Blog posts, pages, media library, SEO metadata
Storefront BigCommerce Catalyst + Makeswift Theme, layout, navigation, design system
Migration Strategy by Tool

Path 1 (Full Migration): Use BC Migration CLI to move all data to BigCommerce. Content migrates to BigCommerce CMS (manual content transfer or CSV import).

Path 2 (Headless WordPress): Use BC Migration CLI for commerce data, Catalyst-WordPress component for content. WordPress stays live as headless CMS.

Path 3 (Hybrid/Phased): Start with Path 2, use BC Migration CLI for commerce, then gradually consolidate content to BigCommerce over 6-12 months.

What This Tool Doesn't Prove (Yet)

This is a technical validation tool. We still need to validate with real customer migrations:

  • Time-to-First-Value: Actual migration timelines with real stores (not just test data)
  • Support Burden: How much hand-holding do merchants need during migration?
  • Edge Case Coverage: What percentage of WooCommerce stores have data patterns this tool can't handle?
  • Post-Migration Issues: What breaks after migration that validation doesn't catch? (SEO redirects, customer login issues, etc.)
  • Agency Adoption: Can WP Engine agency partners run this tool themselves, or does it require BigCommerce Solutions Architect support?
Critical Next Step: First 10 Pilots

What We Need: Deploy BC Migration CLI + Catalyst-WordPress integration to 10 real WP Engine merchants. Half should be Path 1 (full migration), half should be Path 2 (headless).

Success Metrics:

  • 8 of 10 migrations complete successfully within 3-week timeline
  • Zero data loss or corruption incidents
  • Merchant NPS >40 (would recommend to peer)
  • Average time-to-first-value <3 weeks from kickoff to go-live

If these metrics validate: Graduate to Core Product teams and scale. If not: iterate on tooling or kill the initiative.
