Three Migration Paths Validated

The proof-of-concept documents three distinct migration strategies, each optimized for different merchant profiles:

Migration Path Architecture Timeline Best For
Path 1: Full Migration Move all WordPress/WooCommerce content to BigCommerce 4-8 weeks Small stores with basic WordPress usage, minimal content
Path 2: Headless WordPress
✓ Proven in Demo		 WordPress for content, BigCommerce for commerce 2-4 weeks Content-heavy sites, existing WordPress investment
Path 3: Hybrid/Phased Start headless, gradually consolidate to BigCommerce 6-12 months Large enterprises, risk-averse organizations
Critical Insight from POC

Path 2 (Headless WordPress) is likely the fastest path to First 10 customer validation.

Why: Merchants keep their existing WordPress content investment (blogs, pages, SEO rankings) while gaining BigCommerce commerce capabilities. Migration risk is minimal because WordPress remains operational. If something breaks, they can roll back instantly.

This aligns with the CFO's directive: "Looking for something much more basic and simple to manage the cutover." Path 2 is the lowest-risk, fastest-to-value approach.

Technical Architecture

WordPress Content Block Component

The core technical innovation is a Makeswift component that securely fetches and renders WordPress content via REST API.

Security Layer
URL allowlisting, DOMPurify HTML sanitization, SSRF protection, request timeout handling
Performance Layer
Multi-layer caching (server/client), Next.js Image optimization, lazy loading, streaming SSR
Reliability Layer
Error boundaries, retry logic with exponential backoff, graceful degradation, clear error messaging
Developer Experience
TypeScript strict mode, Zod schema validation, comprehensive error logging, visual editor integration

System Architecture

The component follows a secure proxy pattern to prevent direct WordPress API exposure:

  1. Makeswift Editor: Merchant configures WordPress content source (URL, post ID, display options)
  2. Next.js API Route (/api/wordpress): Validates URL against allowlist, fetches content from WordPress REST API
  3. Server-Side Rendering: Content rendered on BigCommerce infrastructure, cached for performance
  4. Client Delivery: Static HTML delivered to user, no direct WordPress API calls from browser
Why This Architecture Matters for WP Engine Partnership

WP Engine hosts 8,000 mid-market WooCommerce stores. Path 2 architecture means these merchants can:

  • Keep their WordPress hosting with WP Engine (revenue retention for WP Engine)
  • Add BigCommerce commerce capabilities without migration risk
  • Gradually consolidate content to BigCommerce over time (Path 3 upgrade path)

Win-Win: WP Engine retains hosting revenue, BigCommerce gains commerce revenue, merchant gets best-of-both-worlds architecture.

How This POC Was Built: AI-Augmented Development

This proof-of-concept demonstrates the AI-augmented development methodology described in the Agentic Engineering Operating Model document. Key practices used:

1. Schema-First Development

Defined WordPress REST API response schemas in TypeScript + Zod before writing any component code. Claude Code generated type-safe API clients, validation logic, and error handling based on schemas.

2. Component Registry Pattern

Built reusable Makeswift components with clear interfaces. AI generates variations (different layout options, styling presets) from base component definitions.

3. Documentation-Driven Development

Wrote merchant-facing documentation first (decision guide, migration playbooks), then built technical implementation to match documented user experience. AI generated technical specs from user stories.

4. Test-First Security

Defined security requirements (URL allowlisting, HTML sanitization, SSRF protection) as test cases. AI implemented security patterns to pass tests, then audited for edge cases.
