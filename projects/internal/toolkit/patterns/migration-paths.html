Migration Paths

Agentic Commerce Toolkit | Reference Pattern

Overview

This playbook provides incremental adoption patterns for organizations migrating from traditional commerce architectures to agentic commerce.

1. Starting Points

1.1 Assess Current State

Current Architecture Migration Complexity Recommended Path
Monolithic Commerce (legacy) High Path A: Strangler Fig
Headless Commerce (MACH) Low Path B: Additive
Hybrid (monolith + some APIs) Medium Path C: API-First
No Commerce (greenfield) N/A Path D: Native Build

1.2 Readiness Assessment

Technical Readiness

  • Commerce APIs available (REST/GraphQL)
  • Customer data accessible via CDP or API
  • Order data accessible via OMS API
  • Payment tokenization supported
  • Event/webhook infrastructure exists

Organizational Readiness

  • Executive sponsorship secured
  • AI/ML skills available or planned
  • Cross-functional team identified
  • Budget approved for MVP

Data Readiness

  • Product catalog structured and accessible
  • Customer consent mechanisms in place
  • FAQ/Knowledge base content exists
  • Order/transaction history accessible

2. Migration Paths

Path A Strangler Fig (Legacy Monolith)

Pattern: Gradually replace legacy functionality with agentic services while maintaining existing systems.

Phase 1: Read-Only Agent +-------------------------------------------------------------+ | Customer -> Agent -> API Gateway -> READ from Legacy DB | | -> NO writes to monolith | +-------------------------------------------------------------+ Phase 2: Parallel Write +-------------------------------------------------------------+ | Customer -> Agent -> API Gateway -> Write to NEW Service | | -> Sync to Legacy (dual) | +-------------------------------------------------------------+ Phase 3: Full Cutover +-------------------------------------------------------------+ | Customer -> Agent -> API Gateway -> New Commerce Stack | | (Legacy retired) | +-------------------------------------------------------------+

12-18 months

Milestones

  1. Month 1-3: API facade over legacy DB for product read
  2. Month 4-6: Service Agent (FAQ/order status) - read-only
  3. Month 7-9: Shopping Agent with cart in new service
  4. Month 10-12: Payment integration, order write
  5. Month 13-18: Migrate remaining functionality, retire legacy

Path B Additive (Headless Commerce)

Pattern: Add agent layer on top of existing headless commerce APIs.

+-------------------------------------------------------------+ | | | +-----------+ +-----------+ +-----------+ | | | Web/App | | Agent | | Marketplace| | | | (existing)| | (NEW) | | (existing) | | | +-----+-----+ +-----+-----+ +-----+-----+ | | | | | | | +----------------+----------------+ | | | | | +--------v--------+ | | | Commerce API | | | | (existing) | | | +-----------------+ | | | +-------------------------------------------------------------+

3-6 months

Milestones

  1. Month 1: AI Platform setup (LLM Gateway, Vector Store)
  2. Month 2: Shopping Agent connected to commerce API
  3. Month 3: Service Agent connected to OMS/CRM
  4. Month 4: Protocol Layer (MCP/ACP)
  5. Month 5-6: Returns Agent, payment mandate

Path C API-First (Hybrid)

Pattern: Build API layer first, then add agents.

Phase 1: API Extraction +-------------------------------------------------------------+ | Monolith -> Extract APIs -> API Gateway -> Existing Channels | +-------------------------------------------------------------+ Phase 2: Agent Addition +-------------------------------------------------------------+ | Agent Layer -> API Gateway -> Extracted APIs + Monolith | +-------------------------------------------------------------+

6-12 months

Milestones

  1. Month 1-3: Extract product, cart, order APIs from monolith
  2. Month 4-5: API Gateway implementation
  3. Month 6-7: AI Platform setup
  4. Month 8-10: Agent development
  5. Month 11-12: Protocol Layer, production launch

Path D Native Build (Greenfield)

Pattern: Build agentic commerce from scratch with modern stack.

6-9 months

Milestones

  1. Month 1: Architecture finalization, team ramp-up
  2. Month 2-3: Core platform (Commerce, AI, Consent)
  3. Month 4-5: Agent development (Shopping, Service)
  4. Month 6-7: Integration (Payment, OMS, CDP)
  5. Month 8-9: Protocol Layer, launch

3.2 Why Start with Service Agent?

Factor Service Agent Shopping Agent
Complexity Lower (read-heavy) Higher (transactional)
Risk Lower (no payments) Higher (payment errors)
Value High (cost savings) High (revenue)
Learning Good foundation Builds on Service
Quick Win Yes (2-3 months) Longer (4-6 months)

Recommendation: Start with Service Agent to build team capability, then expand to Shopping Agent.

4. Data Migration

4.1 Product Data for RAG

1 Export Product Catalog

Source: PIM / Commerce Platform Format: JSON with all attributes Fields: ID, name, description, category, attributes, pricing, availability

2 Generate Embeddings

Process: Batch embedding generation Model: text-embedding-3-large (OpenAI) or similar Chunk: Per product (or per attribute group for large products)

3 Load Vector Store

Destination: Pinecone / Weaviate / Qdrant Metadata: Include category, brand, price range for filtering Index: Refresh on product update (CDC recommended)

4.2 FAQ/Knowledge Base

1 Content Audit

  • Identify all FAQ sources (website, help center, PDFs)
  • Consolidate into single format
  • Update stale content

2 Chunking Strategy

Chunk size: 500-1000 tokens Overlap: 100-200 tokens Metadata: Category, last updated, source URL

3 Continuous Sync

  • Webhook from CMS on content update
  • Re-embed and update vector store
  • Version tracking for rollback

4.3 Customer Data

1 CDP Integration

  • API connection to CDP (Segment, mParticle, etc.)
  • Real-time profile fetch for agent context
  • Consent flag propagation

2 Memory Bootstrap

  • Import last N orders per customer
  • Import saved preferences
  • Clear old/irrelevant data

5.1 Commerce Platform Integration

5.2 Payment Integration

Pattern: Mandate + Token

1. Customer grants mandate (one-time, via trusted UI) -> Mandate stored with scope (amount limit, merchant category) 2. Agent requests payment -> Check mandate validity -> Retrieve SPT (Secure Payment Token) -> Process payment via token (no card details) 3. Audit trail -> Log agent ID, mandate ID, token ID, amount, result

6. Testing Strategy

6.1 Agent Testing Pyramid

+-------------+ | E2E with | (few) | LLM Calls | +-------------+ +---------------------+ | Integration Tests | (some) | (mocked LLM, real | | commerce APIs) | +---------------------+ +-----------------------------+ | Unit Tests | (many) | (tool functions, parsers, | | business logic) | +-----------------------------+

6.2 Conversation Testing

Golden Dataset

  • 100+ representative conversations
  • Cover happy path + edge cases
  • Include escalation scenarios
  • Multi-turn dialogues

Evaluation Metrics

  • Task completion rate
  • Factual accuracy (vs ground truth)
  • Response relevance (LLM-as-judge)
  • Latency P50/P95

Failover behavior 0 dropped Chaos engineering

7. Rollout Strategy

7.1 Progressive Rollout

Stage Timeframe Traffic Details
Stage 1: Internal Week 1-2 Employees only All escalations go to human; intensive monitoring
Stage 2: Beta Week 3-4 5% (opt-in) Prominent feedback mechanism; daily review
Stage 3: Limited GA Week 5-8 25% Automated escalation; A/B test vs non-agent
Stage 4: Full GA Week 9+ 100% Default for eligible queries; continuous optimization

7.2 Feature Flags

Flag Purpose Default
agent.enabled Master kill switch true
agent.shopping.enabled Shopping agent true
agent.checkout.enabled Agent-initiated checkout false (Phase 2)
agent.payment.enabled Agent payment processing false (Phase 2)
agent.confidence.threshold Escalation threshold 0.7
agent.max.order.value Maximum order without human $500

7.3 Rollback Procedures

Automated Rollback Triggers

  • Error rate > 5% (5-minute window)
  • Latency P95 > 10s
  • Customer escalation rate > 40%
  • Payment failure rate > 2%

Manual Rollback

# Disable agent via feature flag curl -X POST /api/flags/agent.enabled -d '{"value": false}' # Or route all traffic to fallback kubectl rollout undo deployment/agent-service

9. Common Pitfalls & Mitigations

Pitfall Mitigation
Over-scoping MVP Start with single agent (Service), prove value, expand
Ignoring latency Set latency budgets early; optimize RAG, cache aggressively
Poor prompt iteration Prompt registry + A/B testing from Day 1
Insufficient observability LangSmith/Phoenix mandatory before production
Underestimating content quality Audit FAQ/KB before RAG; garbage in = garbage out
No escalation path Human handoff mandatory; measure and reduce over time
Payment security gaps Mandate + SPT pattern; never expose credentials
Ignoring edge cases Golden dataset with edge cases; expand continuously

10. Checklist: Go-Live Readiness

Technical Readiness

  • All critical path components deployed
  • Load testing completed at 2x expected traffic
  • Failover tested (LLM, database, payment)
  • Observability dashboards operational
  • Alerting configured and tested
  • Rollback procedures documented and tested

Operational Readiness

  • On-call rotation established
  • Runbooks created for common issues
  • Escalation paths defined
  • Support team trained on agent behavior
  • Customer communication prepared

Compliance Readiness

  • Privacy review completed
  • Security review completed
  • Consent mechanisms verified
  • Audit logging operational
  • Data retention policies applied

Business Readiness

  • Success metrics defined and tracked
  • Stakeholder communication complete
  • Feedback collection mechanism ready
  • Phase 2 roadmap approved
