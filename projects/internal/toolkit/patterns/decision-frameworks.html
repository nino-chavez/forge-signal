ADR-001: Multi-Protocol Support vs Single Protocol

Accepted
Context

Multiple LLM platforms (OpenAI, Anthropic, Google) are introducing different protocols for agent-commerce interaction: ACP, MCP, and AP2. We need to decide whether to support one protocol or multiple.

Decision

Support all three protocols (ACP, MCP, AP2) through a Protocol Adapter layer that normalizes external requests to a unified internal API.

Consequences
  • (+) Maximum agent reach across platforms
  • (+) No vendor lock-in
  • (+) Future-proof as standards evolve
  • (-) Higher initial integration complexity
  • (-) Ongoing maintenance for protocol updates
Alternatives Considered
  1. MCP-only (open standard) — Rejected: would miss GPT-4 users
  2. ACP-only (highest volume) — Rejected: vendor lock-in risk
  3. Custom protocol — Rejected: no platform adoption

ADR-002: Consent Layer Architecture

Accepted
Context

External agents requesting customer data create compliance risks (GDPR, CCPA). We need a mechanism to verify agent identity and enforce data access policies.

Decision

Implement a dedicated Consent & Data Access Control layer between the Protocol Layer and API Gateway, including:

  • Agent identity verification
  • Data classification tags on all responses
  • Consent propagation from CDP
  • Comprehensive audit logging
Consequences
  • (+) Regulatory compliance by design
  • (+) Clear audit trail for all agent data access
  • (+) Granular control over data exposure
  • (-) Additional latency (~20-50ms per request)
  • (-) Complexity in consent flow management
Alternatives Considered
  1. Rely on platform consent (e.g., ChatGPT's user consent) — Rejected: insufficient control
  2. Embed in API Gateway — Rejected: too complex for single layer
  3. Per-service consent checks — Rejected: inconsistent enforcement

ADR-003: Agent Payment Authorization Model

Accepted
Context

Agents need to complete purchases on customer behalf, but exposing full payment credentials to agents is unacceptable. We need a secure payment delegation model.

Decision

Implement a Mandate-based payment system with:

  1. Mandate Manager: Captures customer authorization with scope limits
  2. Secure Payment Tokens (SPT): Tokenized payment methods
  3. Token Lifecycle: Issuance, usage tracking, expiration
  4. Payment Audit Log: Complete trail of agent-initiated payments
Consequences
  • (+) Secure payment handling without credential exposure
  • (+) Customer control via mandate limits
  • (+) Auditable for disputes
  • (-) Requires customer onboarding to create mandates
  • (-) Additional complexity vs direct payment
Alternatives Considered
  1. Full credential access with scope — Rejected: security risk
  2. Redirect to merchant checkout — Rejected: breaks agent flow
  3. Platform-managed payments only — Rejected: limited to platform users

ADR-004: LLM Provider Strategy

Accepted
Context

Multiple LLM providers offer different strengths (GPT-4 for reasoning, Claude for long context, Gemini for multimodal). We need to decide on provider strategy.

Decision

Implement an LLM Gateway with:

  • Multi-provider support (GPT-4, Claude, Gemini)
  • Intelligent routing based on task type
  • Fallback chain for resilience (GPT-4 → Claude → Gemini → Local → Cache)
  • Cost optimization through provider selection
Consequences
  • (+) Best-in-class for each task type
  • (+) Resilience through redundancy
  • (+) Cost optimization opportunities
  • (-) Complexity in managing multiple providers
  • (-) Prompt tuning per provider
Alternatives Considered
  1. Single provider (GPT-4 only) — Rejected: single point of failure
  2. Local models only — Rejected: capability limitations
  3. Provider per use case (fixed) — Rejected: no failover

ADR-005: Memory and Context Persistence

Accepted
Context

Agents need to remember customer context across sessions to provide personalized experiences ("You bought X last month"). Current LLM context windows are insufficient for long-term memory.

Decision

Implement a dedicated Memory Store with:

  • Session memory: Current conversation context
  • Short-term memory: Recent interactions (7 days)
  • Long-term memory: Customer preferences, purchase patterns
  • Semantic indexing: Vector-based retrieval
Consequences
  • (+) Personalized agent experiences
  • (+) Reduced repetition in conversations
  • (+) Context for complex multi-turn interactions
  • (-) Data storage and privacy considerations
  • (-) Memory relevance decay management
Alternatives Considered
  1. Rely on LLM context window — Rejected: limited to ~128K tokens
  2. CDP only — Rejected: not conversation-aware
  3. No persistent memory — Rejected: poor experience

ADR-006: Domain Agent Architecture

Accepted
Context

We need to decide whether to build a single "mega-agent" or specialized domain agents for different commerce functions.

Decision

Implement specialized Domain Agents:

  • Shopping Agent: Discovery, comparison, cart
  • Service Agent: Support, FAQ, warranties
  • Returns Agent: RMA, refunds, exchanges
  • Fulfillment Agent: Tracking, delivery
  • Pricing Agent: Dynamic pricing, promotions

Agents communicate via orchestration layer (LangGraph) with clear handoff protocols.

Consequences
  • (+) Separation of concerns
  • (+) Independent scaling and iteration
  • (+) Specialized prompt optimization
  • (+) Clear ownership boundaries
  • (-) Inter-agent coordination complexity
  • (-) Potential context loss during handoff
Alternatives Considered
  1. Single mega-agent — Rejected: unmaintainable prompt, no specialization
  2. Microservice per intent — Rejected: too granular, coordination overhead
  3. Vendor-specific agents — Rejected: lock-in, inconsistent experience

ADR-007: Observability Strategy

Accepted
Context

Traditional APM tools are insufficient for LLM-based systems. We need specialized observability for tracing, evaluation, and cost tracking.

Decision

Implement LLM-specific observability using LangSmith or Phoenix:

  • Request-level tracing (prompt → response)
  • Token usage and cost tracking
  • Quality evaluation (automated + human)
  • Latency breakdown by component
  • A/B testing for prompts
Consequences
  • (+) Debuggable LLM interactions
  • (+) Cost visibility and optimization
  • (+) Quality measurement over time
  • (-) Additional instrumentation overhead
  • (-) Observability platform cost
Alternatives Considered
  1. Traditional APM only — Rejected: no LLM-specific insights
  2. Custom logging — Rejected: reinventing the wheel
  3. No observability — Rejected: unacceptable for production

ADR-008: Human-in-the-Loop Design

Accepted
Context

Agents will encounter situations where they shouldn't proceed autonomously (low confidence, high-value transactions, frustrated customers). We need a structured escalation path.

Decision

Implement Trust & Escalation domain with:

  • Confidence Scoring: ML-based uncertainty detection
  • Human Handoff Protocol: Seamless transfer to live support
  • Guardrails: Policy enforcement before response
  • Approval Workflows: For high-value operations

Escalation triggers:

  • Confidence < 0.7
  • Order value > $500
  • Mandate exceeded
  • Customer frustration detected
  • Explicit escalation request
Consequences
  • (+) Safety net for agent failures
  • (+) Customer satisfaction preservation
  • (+) Liability management
  • (-) Operational cost for human support
  • (-) Latency in escalated scenarios
Alternatives Considered
  1. Full automation — Rejected: unacceptable risk
  2. Human approval for all — Rejected: defeats automation value
  3. Fixed thresholds only — Rejected: no adaptability

ADR-009: Structured Data for GEO

Accepted
Context

AI search engines (Perplexity, Google AI Overviews) source information from structured data. We need to optimize for Generative Engine Optimization (GEO).

Decision

Implement comprehensive Schema.org markup:

  • Product schema with all attributes
  • FAQ schema for common questions
  • Review/Rating aggregates
  • Offer/Price markup
  • Organization and brand identity

Add GEO-specific content:

  • Buying guides
  • Comparison articles
  • How-to content
  • Expert reviews (E-E-A-T)
Consequences
  • (+) AI search visibility
  • (+) Rich snippets in traditional search
  • (+) Source citations from AI engines
  • (-) Content creation investment
  • (-) Ongoing content freshness management
Alternatives Considered
  1. Ignore AI search — Rejected: growing traffic source
  2. Schema only — Rejected: insufficient for citations
  3. Content only — Rejected: no structured extraction

ADR-010: B2C Focus with B2B Extensibility

Accepted
Context

The architecture must balance immediate B2C retail needs with potential B2B commerce extension.

Decision

Design architecture with B2C focus:

  • Consumer checkout flows
  • Individual customer identity
  • Standard payment methods
  • Simple cart/checkout

Add B2B extension points:

  • Customer vs Account abstraction
  • Quote workflow hooks
  • Approval workflow (reusable from Trust domain)
  • Contract pricing placeholders

Create separate B2B variant architecture document if B2B becomes priority.

Consequences
  • (+) Focused B2C delivery
  • (+) Clear extension path
  • (+) Reusable Trust domain patterns
  • (-) B2B not immediately supported
  • (-) May require refactoring for B2B
Alternatives Considered
  1. Full B2B+B2C from start — Rejected: scope creep
  2. B2C only, no extension — Rejected: limits future
  3. B2B first — Rejected: not primary use case
